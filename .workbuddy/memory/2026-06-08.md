# 2026-06-08

## design-craft skill 更新

- 新增 **阶段 2.3：交互对象总览**，位于阶段 2（语义分析）之后、阶段 2.5（需求拆分）之前
- 目的：在拆分需求和设计之前，先明确对象/模块/用户间的调用关系，建立协作地图
- 三个确认点：外部系统作为独立节点、不确定关系直接询问用户、不需要传递到后续阶段
- 核心原则新增第4条"交互先于设计"
- 回退机制新增"交互对象回退"（阶段 2.3 专属）
- 成熟度高的场景跳转目标从阶段 2.5 改为阶段 2.3
- 快速通道仍保留阶段 2.3

## interaction-design skill 新建

- 定位：requirement-mining → **interaction-design** → design-craft
- 设计人机交互层，不画 UI，不设计技术架构
- 场景自适应：阶段 1 识别场景类型（脚本/CLI、Web、API、SDK、其他），后续阶段自动适配术语和输出格式
- 7 个阶段：角色与场景类型 → 场景梳理 → 交互流程 → 反馈与状态 → 异常与边界 → 交互约束 → 落盘输出
- 每阶段必须用户确认后才继续

## work-breakdown skill 新建

- 定位：requirement-mining → interaction-design → **work-breakdown** → 每个切片独立走 design-craft
- 参考 mattpocock/skills 的 to-issues（垂直切片 + AFK/HITL 分类）
- 核心概念：垂直切片，每个切片贯穿所有技术层，完成即可独立验证
- 6 个阶段：输入分析 → 分层识别 → 垂直切片拆分 → 独立性校验 → 粒度确认 → 输出清单
- 与 design-craft 的关系：work-breakdown 在设计之前，拆完后每个切片独立走 design-craft

## data-flow-model skill 新建

- 定位：work-breakdown → **data-flow-model** → design-craft
- 两层结构：通用层（ER 图 + 数据流图）+ 场景层（按类型叠加分析）
- 场景类型：简单 CRUD、并发、分布式、实时流、批处理
- 6 个阶段：场景类型识别 → 实体识别 → ER 图 → 数据流图 → 场景特定分析 → 落盘输出
- 选择 ER + 数据流方向方案（而非 DFD 或架构图），因为学习成本最低、适用范围最广

## demo-verify skill 新建

- 定位：design-craft → **demo-verify** → 完整开发
- 复杂需求自动触发，简单需求跳过
- 5 个阶段：风险识别 → 验证范围 → 实现原型 → 验证评估 → 决策与清理
- 原型代码放 `.demo-verify/` 独立目录，开发完成后删除
- 结论：通过/有条件通过/不通过，不通过则回退设计

## requirement-mining skill 更新

- 新增 **Step 8：落盘归档**，在 Step 7 输出报告后自主提示用户是否持久化
- 存储位置可自定义，默认建议 `.requirements/`
- 配置文件：`.requirements/config`
- 文件命名：`{YYYY-MM-DD}-{feature-name}.md`
- 仅在用户确认后才写入文件
- **持久化格式为结论性摘要**（非完整报告），包含：核心诉求、需求形态、核心场景、根本性分析结论、需求清单、关键假设、非功能性约束、潜在风险
- 下游 skill（design-craft、code-review、challenger）可引用持久化的需求文档

## design-craft skill 更新（存储配置）

- 阶段 5（落盘归档）增加存储位置逻辑：读取 `.requirements/config`，未配置则询问用户
- 设计文档存放在 `{storage_path}/{date}-{feature}/design/`

## implementation-report skill 新建

- 定位：需求完成后生成实现总结报告
- 与需求摘要一一对应
- 报告内容：需求回顾、设计概要、实现结果、偏差记录、关联提交、验收对照、遗留问题
- 自动更新需求摘要的 status/commits/report 字段
- 使用 `.requirements/config` 中的存储配置

## 存储结构调整

- 需求摘要：`requirement.md`
- 设计文档：`design/` 子目录
