# 需求挖掘报告：document-writer Skill

## 1. 原始需求描述

> 创建撰写 README 及其相关子文档的 skill。AI 写的 README 太简略，只有使用命令，没有展示信息示例。不同功能的项目应使用不同的编写策略。README 作为索引枢纽，详细内容拆分到子文档。

## 2. 需求澄清

### 2.1 需求形态

伪装成需求的解决方案 → 还原后的真实需求：系统化地让 AI 产出的项目文档具备信息密度和场景适配力，而非套模板式的空壳文档。

### 2.2 功能本质

消除 AI 生成项目文档时「有骨架无血肉」的通病 —— 只有安装命令，没有展示项目实际效果、输入输出示例、边界情况的实质性内容。

### 2.3 使用场景与角色

| # | 场景 | 角色 | 触发条件 | 期望产出 |
|---|------|------|----------|----------|
| 1 | 新建开源库 | 开发者 | 代码写完，需要对外发布 | README + API 文档，每个 API 有输入输出示例 |
| 2 | CLI 工具发布 | 工具作者 | 命令行工具开发完成 | README 含命令演示 + 终端输出示例 |
| 3 | Web 应用/服务 | 全栈开发者 | 项目需要交付或开源 | README + 部署文档 + 架构说明 |
| 4 | SDK/内部库 | 平台开发者 | 内部发布给其他团队 | README + 使用指南 + 快速集成示例 |
| 5 | 已有项目补文档 | 项目接手者 | 接手无文档的老项目 | 逆向生成 README，从代码反推使用方式 |

- 场景关联性：场景 1-4 是新建项目时的并行分支，场景 5 是补救场景（逆向工程），与 1-4 互斥。

### 2.4 核心痛点

- AI 写的 README 像填空题，真正体现项目价值的示例和效果展示几乎为零
- 不同项目类型需要不同策略，但 AI 不分青红皂白用一套模板
- 子文档要么缺失，要么同样空洞

### 2.5 期望体验

告诉 skill 「这是一个 React 组件库」，它就知道该重点展示组件示例和 Props 表格；告诉它是 CLI 工具，就知道该展示终端输出效果 —— 每种项目类型有对应的信息展示策略。

### 2.6 深层动机

文档是项目的第一印象，空洞的 README 直接劝退潜在使用者。需要的不只是「能生成文档」，而是「生成的文档有说服力」。

## 3. 根本性分析

### 3.1 核心问题

AI 生成的项目文档存在系统性缺陷：信息密度低、项目类型无感知、文档组织无策略。

### 3.2 根因链

```
AI 缺乏项目类型感知 → 不知道该项目的读者关心什么 → 不知道该重点展示什么信息
→ 只能套通用模板填空 → 产出的文档空洞、无说服力

同时 AI 缺乏文档组织策略 → 要么全塞在一个 README 里（臃肿）
→ 要么无脑拆分子文档（碎片化） → 读者找不到想要的信息
```

### 3.3 方案评估

**判断**：情况 A（方案对症）

**理由**：根因是「缺少类型感知的文档策略编码」，skill 是最适合承载这类策略知识的载体。

### 3.4 预期效果分析

| 维度 | 评估 |
|------|------|
| 核心场景覆盖度 | 高 - 库、CLI、Web 应用、插件四类覆盖绝大多数 |
| 痛点解决程度 | 高 - 类型感知策略 + 示例强制要求，从机制上杜绝空洞 |
| 用户体验提升 | 从「生成后大量手改」变为「生成即可用」 |
| 潜在副作用 | 低 - 策略设计不当可能导致过度拆分，已内置阈值控制 |

## 4. 需求清单

### 4.1 需求拆分清单

| 优先级 | 需求 ID | 需求描述 | 预期效果 | 依赖 | 验收标准 |
|--------|---------|----------|----------|------|----------|
| P0 | REQ-01 | 项目类型自动识别 | 无需手动声明类型，skill 自动判断 | - | 四类项目识别准确率 ≥ 90% |
| P0 | REQ-02 | 类型专属编写策略 | 不同项目类型产出的文档结构不同，对齐读者关注点 | REQ-01 | 库类型必含 API 示例，CLI 必含命令输出示例 |
| P0 | REQ-03 | README 作为索引枢纽 | README 一屏内可读完概览，细节通过链接跳转 | - | 每个子文档链接准确对应 |
| P0 | REQ-04 | 示例强制规则 | 每个对外接口必须有输入→输出完整示例 | - | 任意接口 100% 附有可运行示例 |
| P1 | REQ-05 | 拆分阈值判断 | 简单项目不冗余拆分，复杂项目不缺文档 | - | 简单库（≤3 API）不生成子文档 |
| P1 | REQ-06 | 子文档自动生成 | 子文档组织清晰，职责边界明确 | REQ-03, REQ-05 | 子文档间无内容重复 |
| P1 | REQ-07 | 内容质量守护规则 | 杜绝空洞内容 | - | 全文无占位符文本 |
| P2 | REQ-09 | 逆向生成模式 | 快速补老项目文档 | - | 能从代码提取 ≥80% 公开接口 |

### 4.2 需求依赖图

```
REQ-01 → REQ-02 → REQ-03 + REQ-04 → REQ-05 + REQ-06 → REQ-07
                                                              ↓
REQ-09（独立分支）                                         全流程约束
```

### 4.3 项目类型策略矩阵

| 项目类型 | 识别特征 | README 核心章节 | 触发拆分的子文档 |
|----------|----------|-----------------|------------------|
| 库/SDK | package.json 含 main/exports，setup.py，go.mod | 概述→快速开始→最简示例→子文档导航 | API Reference、Configuration、Advanced Usage |
| CLI 工具 | bin 字段，argparse/commander/cobra 等框架 | 概述→安装→核心命令演示→子文档导航 | Command Reference、Configuration、Integration |
| Web 应用/服务 | express/next/flask/gin 等框架，含路由定义 | 概述→快速启动→架构概览→子文档导航 | Architecture、Deployment、Development Guide |
| 插件/扩展 | 依赖宿主框架 | 概述→安装→最简配置示例→子文档导航 | Configuration、API/Hooks、Examples |

### 4.4 信息密度底线

| 章节 | 禁止出现 | 必须出现 |
|------|---------|---------|
| 概述 | 仅一句话 | 做什么 + 解决什么问题 + 和其他方案的区别 |
| 安装 | 仅 npm install | 完整命令 + 前置条件 + 安装后验证方法 |
| 快速开始 | 仅代码片段 | 完整可运行示例，含输入和预期输出 |
| 子文档导航 | 无 | 每个链接附说明「这个文档里有什么」 |

### 4.5 成功度量指标

| 度量维度 | 指标 | 基线 | 目标 |
|----------|------|------|------|
| 信息密度 | 有实质示例的章节占比 | ~20% | ≥ 80% |
| 类型适配 | 策略选择准确率 | 0% | ≥ 90% |
| 拆分合理性 | 过度拆分/不足率 | 无策略 | ≤ 15% |
| 空洞内容 | 占位符出现率 | 常见 | 0% |

## 5. 潜在风险与注意事项

| 风险 | 说明 | 缓解方式 |
|------|------|----------|
| 类型误判 | 混合项目可能被错误分类 | 允许用户手动覆盖类型选择 |
| 示例失真 | 从代码推断的示例可能与实际行为不符 | 标注「从代码推断」，建议用户验证 |
| 拆分过度 | 小项目拆出过多子文档 | REQ-05 阈值可调，允许用户拒绝拆分 |
