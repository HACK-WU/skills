# 示例：用户注册功能测试计划

假设 requirement.md 中有以下需求条目：

```
| 优先级 | 需求 ID | 需求描述 | 验收标准 |
|--------|---------|----------|----------|
| P0 | REQ-01 | 用户可以通过邮箱注册账号 | 用户可以正常完成注册流程，收到验证邮件 |
| P0 | REQ-02 | 注册流程步骤减少 50% | 从原来的 10 步减少到 5 步以内 |
| P1 | REQ-03 | 接口响应时间 < 200ms | 注册接口平均响应时间 < 200ms |
| P0 | REQ-04 | 无权限用户不能访问管理后台 | 普通用户无法访问 /admin 路径 |
| P1 | REQ-05 | 用户密码加密存储 | 密码使用 bcrypt 哈希存储，非明文 |
| P0 | REQ-06 | 注册数据持久化到数据库 | 用户信息写入数据库，断电不丢失 |

潜在风险：
- 高并发注册时数据库可能成为瓶颈
- 验证邮件可能被邮箱服务商拦截
```

---

## Before（典型 AI 默认输出）

```markdown
# 测试计划

## 测试用例

### 1. 注册功能测试
- 测试正常注册流程
- 测试邮箱格式验证
- 测试密码强度验证
- 测试重复注册

### 2. 性能测试
- 测试接口响应时间
- 测试并发注册

### 3. 安全测试
- 测试密码加密
- 测试权限控制

### 4. 数据测试
- 测试数据持久化
```

**问题诊断**：
- 只有标题，没有具体步骤和预期结果
- 没有追溯到具体 REQ 条目
- 没有区分测试策略（功能型/度量型/性能型混在一起）
- 没有覆盖矩阵，无法确认覆盖率
- 没有前置条件和通过标准
- 没有边界条件和异常路径

---

## 核心差异总结

| 维度 | AI 默认输出 | 本 skill 输出 |
|------|------------|--------------|
| 具体程度 | 只有标题 | 完整的前置条件 + 步骤 + 预期结果 + 通过标准 |
| 需求追溯 | 无 | 每个用例标注关联 REQ |
| 策略区分 | 混在一起 | 按功能/度量/性能/否定/安全/数据分类 |
| 边界条件 | 无 | 每个功能型用例有异常路径和边界条件 |
| 覆盖率 | 不可知 | 明确的覆盖矩阵 |
| 风险测试 | 无 | 从 requirement.md 的风险项提取补充用例 |
