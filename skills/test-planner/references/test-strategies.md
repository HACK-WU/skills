# 测试策略定义

本文档定义验收标准文本特征到测试策略的映射规则，以及各策略下的用例生成模板。

## 策略选择规则

### 关键词匹配表

从验收标准原文中匹配以下关键词（不区分大小写），命中即选择对应策略。可多选。

| 策略类型 | 匹配关键词（中文） | 匹配关键词（英文） | 正则模式 |
|----------|-------------------|-------------------|----------|
| **功能型** | 可以、能够、支持、正常、完成、实现、提供、允许 | can, support, allow, enable, work, function, provide | `(可以|能够|支持|正常|完成)` |
| **否定型** | 不能、禁止、不可、拒绝、阻止、防止、避免 | cannot, must not, reject, block, prevent, deny | `(不能|禁止|不可|拒绝|阻止|防止)` |
| **度量型** | 减少、提升、达到、至少、不超过、大于、小于、百分比 | reduce, increase, reach, at least, no more than, ≥, ≤ | `\d+\s*[%％]` 或 `(减少|提升|达到)\s*\d+` |
| **性能型** | 响应时间、延迟、并发、吞吐、超时、毫秒、秒内 | latency, response time, concurrent, throughput, timeout | `(\d+)\s*(ms|毫秒|秒|s)` |
| **安全型** | 权限、加密、认证、授权、防护、脱敏、审计 | permission, encrypt, auth, authorize, protect, audit | `(权限|加密|认证|授权|防护)` |
| **数据型** | 数据、记录、持久化、同步、备份、一致、存储、写入 | data, record, persist, sync, backup, consistent | `(数据|记录|持久化|同步|备份|一致)` |
| **体验型** | 步骤减少、满意度、学习成本、易用、操作简化、一键 | step, satisfaction, usability, easy, simplify | `(步骤|满意度|学习成本|易用|一键)` |

### 优先级规则

当多个策略同时匹配时：

1. **安全型** 永远优先（安全问题不可妥协）
2. **否定型** 优先于功能型（否定型是功能型的反向验证）
3. **性能型** 优先于度量型（性能是特殊的度量）
4. 其余策略并行执行

## 各策略用例模板

### 功能型

验收标准示例：「用户可以正常登录系统」

**生成 3 类用例**：

```
TC-{ID}-01：正常路径 —— 正确凭据登录
  前置条件：用户已注册，账号状态正常
  步骤：
    1. 打开登录页面
    2. 输入正确的用户名和密码
    3. 点击登录按钮
  预期结果：登录成功，跳转到首页
  通过标准：页面 URL 变为首页，显示用户名

TC-{ID}-02：异常路径 —— 错误凭据登录
  前置条件：用户已注册
  步骤：
    1. 打开登录页面
    2. 输入正确的用户名和错误的密码
    3. 点击登录按钮
  预期结果：登录失败，显示错误提示
  通过标准：页面显示「密码错误」，不跳转

TC-{ID}-03：边界条件 —— 空输入
  前置条件：无
  步骤：
    1. 打开登录页面
    2. 不输入任何内容
    3. 点击登录按钮
  预期结果：提示必填项
  通过标准：用户名和密码输入框显示必填提示
```

### 否定型

验收标准示例：「无权限用户不能访问管理后台」

**生成 2-3 类用例**：

```
TC-{ID}-01：越权访问 —— 普通用户访问管理页面
  前置条件：以普通用户角色登录
  步骤：
    1. 直接在浏览器地址栏输入管理后台 URL
    2. 访问
  预期结果：被拦截，返回 403 或跳转到无权限页面
  通过标准：不显示管理后台内容

TC-{ID}-02：未认证访问 —— 不携带 Token 访问
  前置条件：未登录状态
  步骤：
    1. 直接访问受保护的 API 接口（不携带 Authorization 头）
  预期结果：返回 401
  通过标准：响应状态码为 401

TC-{ID}-03：伪造凭证 —— 使用过期/篡改的 Token
  前置条件：有一个已过期的 Token
  步骤：
    1. 使用过期 Token 访问受保护接口
  预期结果：返回 401
  通过标准：响应状态码为 401，提示 Token 无效
```

### 度量型

验收标准示例：「注册流程步骤减少 50%」

**生成 2 类用例**：

```
TC-{ID}-01：基线测量 —— 记录当前步骤数
  前置条件：使用旧版本或现有流程
  步骤：
    1. 记录完成注册的完整操作步骤数
    2. 每个用户操作（点击、输入、提交）计为 1 步
  预期结果：记录基线值（如 10 步）
  通过标准：基线值记录完整

TC-{ID}-02：目标验证 —— 新流程步骤数
  前置条件：使用新版本
  步骤：
    1. 记录完成注册的完整操作步骤数
    2. 对比基线值
  预期结果：步骤数 ≤ 基线值 × 50%
  通过标准：步骤数 ≤ 5 步（基线 10 步的 50%）
```

### 性能型

验收标准示例：「接口响应时间 < 200ms」

**生成 3 类用例**：

```
TC-{ID}-01：单用户响应时间
  前置条件：服务正常运行，网络延迟 < 10ms
  步骤：
    1. 发送单次 API 请求
    2. 记录从发送到收到完整响应的时间
    3. 重复 10 次取平均值
  预期结果：平均响应时间 < 200ms
  通过标准：P50 < 150ms, P99 < 200ms

TC-{ID}-02：并发响应时间
  前置条件：服务正常运行
  步骤：
    1. 使用压测工具（如 wrk/ab/k6）发送 100 并发请求
    2. 持续 30 秒
    3. 记录 P50/P95/P99 响应时间和错误率
  预期结果：P99 < 200ms，错误率 < 1%
  通过标准：满足上述指标

TC-{ID}-03：持续负载稳定性
  前置条件：服务正常运行
  步骤：
    1. 使用 50 并发持续请求 10 分钟
    2. 监控内存使用和响应时间趋势
  预期结果：无内存泄漏，响应时间无明显退化
  通过标准：结束时响应时间不超过开始时的 150%
```

### 安全型

验收标准示例：「用户密码加密存储」

**生成 2-3 类用例**：

```
TC-{ID}-01：密码存储验证
  前置条件：已注册用户
  步骤：
    1. 查询数据库中该用户的密码字段
    2. 验证存储格式（应为哈希值，非明文）
  预期结果：密码为 bcrypt/scrypt/argon2 哈希值
  通过标准：密码字段非明文，且包含哈希算法标识

TC-{ID}-02：传输加密验证
  前置条件：抓包工具就绪
  步骤：
    1. 在登录时使用抓包工具捕获请求
    2. 检查请求体中密码是否为明文
  预期结果：请求使用 HTTPS，密码在传输中不可见
  通过标准：HTTP 请求中不包含明文密码
```

### 数据型

验收标准示例：「用户数据持久化到数据库」

**生成 2-3 类用例**：

```
TC-{ID}-01：写入验证
  前置条件：数据库可访问
  步骤：
    1. 通过接口创建一条用户记录
    2. 直接查询数据库验证记录存在
  预期结果：数据库中存在对应记录，字段值正确
  通过标准：所有字段值与输入一致

TC-{ID}-02：读取一致性
  前置条件：已写入数据
  步骤：
    1. 通过接口读取该记录
    2. 对比写入时的值
  预期结果：读取值与写入值完全一致
  通过标准：所有字段值无偏差

TC-{ID}-03：异常恢复
  前置条件：模拟数据库连接中断
  步骤：
    1. 在写入过程中断开数据库连接
    2. 恢复连接后重新读取
  预期结果：未完成的写入被回滚，数据一致性不受影响
  通过标准：数据库中不存在半写入的记录
```

### 体验型

验收标准示例：「首次使用无需阅读文档即可完成基本操作」

**生成 2 类用例**：

```
TC-{ID}-01：无引导操作测试
  前置条件：新用户首次使用，不提供任何文档
  步骤：
    1. 打开应用
    2. 尝试完成核心操作（如创建第一个项目）
    3. 记录是否需要外部帮助
  预期结果：能在 5 分钟内完成
  通过标准：无需查看文档即可完成

TC-{ID}-02：操作步骤验证
  前置条件：用户熟悉基本操作
  步骤：
    1. 完成核心操作
    2. 记录操作步骤数和耗时
  预期结果：步骤数和耗时符合验收标准
  通过标准：步骤数 ≤ 预期值
```

## 项目类型调整

### 库/SDK 调整

- 功能型用例改为 API 调用示例 + 返回值断言
- 增加**向后兼容测试**：验证 API 签名未破坏性变更
- 增加**类型安全测试**（TypeScript）：类型推断是否正确

### CLI 工具调整

- 功能型用例改为终端命令 + stdout 断言
- 增加 **exit code 测试**：成功为 0，失败为非 0
- 增加 **stderr 测试**：错误信息输出到 stderr
- 增加**管道组合测试**：与其他 CLI 工具的管道组合

### Web 应用调整

- 功能型用例改为 E2E 用户流程
- 增加**浏览器兼容测试**：Chrome/Firefox/Safari
- 增加**响应式测试**：移动端/平板/桌面
- 增加**无障碍测试**：键盘导航、屏幕阅读器

### 插件/扩展调整

- 功能型用例改为宿主环境中配置 + 使用
- 增加**宿主版本兼容测试**：最低版本到最新版本
- 增加**配置组合测试**：不同配置项的排列组合
