# 设计文档评审器参考资料

## 一、格式识别示例

### ✅ design-craft 格式识别信号

```markdown
> 状态：草案

# 需求背景 & 目标
...

# 关键环节一览图
\```mermaid
flowchart LR
    ...
\```

# 总体方案设计
## 共享术语速查
...
```

**识别依据**：状态 metadata + 一览图 mermaid 块 + 共享术语速查 → design-craft 格式，加载精准基线。

### ❌ 自由格式

```markdown
# 用户管理模块设计

## 背景
我们需要一个用户管理模块...

## 方案
使用 MySQL 存储...
```

**识别依据**：无 design-craft 特征信号 → 自由格式，降级为通用规则评审。

## 三、内容质量检查示例

### ❌ 现状描述不合格

**文档原文**：
> 当前代码结构不够清晰，维护成本高。

**评审意见**：
```text
🟡 警告 | S-01 §2 | 现状描述缺少具体文件路径或代码行号
→ 修改建议：补充涉及的源文件路径（如 `src/import-kb.ts`）和具体痛点（如"import-kb.ts L120-180 的 7 步流程散布在 3 个函数中，修改任一步骤需改动 2+ 处"）
```

### ❌ 决策点否决理由写空话

**文档原文**：

| 决策 | 选择 | 理由 | 备选方案 | 否决原因 |
|------|------|------|---------|---------|
| 存储方案 | Redis | 性能好 | SQLite | 不合适 |

**评审意见**：
```text
🔴 阻断 | S-02 §3.2 | 决策点"存储方案"否决理由"不合适"不具体
→ 修改建议：改为具体技术理由，如"SQLite 单文件锁竞争，高并发写入延迟 >50ms，不满足 P99 <10ms 的延迟要求"
```

### ❌ 非功能性需求假设未标注

**文档原文**：
> 方案采用单机内存缓存，启动时全量加载。

**评审意见**：
```text
🟡 警告 | S-01 §3 | 方案隐含"单机低量级"假设但未显式标注
→ 修改建议：在方案章开头标注"本方案默认单机低量级（<1万条），若数据量超过此范围需切换为分布式缓存"
```

## 四、技术正确性挑战示例

### 高置信度挑战

**文档原文**：
> 使用 `SELECT * FROM users WHERE id = ?` 加行锁实现互斥。

**评审意见**：
```text
🔴 阻断 | S-03 §3 | [置信度：高] SELECT ... FOR UPDATE 在读已提交隔离级别下无法防止幻读，若用户并发创建可能导致重复记录
→ 修改建议：改用唯一索引 + INSERT ON DUPLICATE KEY，或在可重复读隔离级别下使用 SELECT ... FOR UPDATE
→ 依据：MySQL 8.0 文档，RC 隔离级别下行锁不锁间隙
```

### 中置信度挑战

**文档原文**：
> 使用 Redis SETNX 实现分布式锁，超时时间 30 秒。

**评审意见**：
```text
🟡 警告 | S-02 §3 | [置信度：中] 30 秒超时可能不够：若业务逻辑执行超过 30 秒，锁会自动释放导致并发问题
→ 修改建议：评估业务最长执行时间，或改用 Redisson 看门狗机制自动续期
→ 依据：分布式锁超时是常见陷阱，但具体业务执行时间需确认
```

### 低置信度挑战

**文档原文**：
> 使用消息队列异步处理订单。

**评审意见**：
```text
🟢 建议 | S-04 §3 | [置信度：低] 消息队列消费失败时的重试策略是否需要考虑？
→ 修改建议：如有必要，补充消费失败的重试次数和死信队列处理策略
→ 依据：通用最佳实践，但文档可能已在其他章节覆盖
```

## 五、一致性校验示例

### ❌ 术语同名异义

**S-01 术语表**：
| 术语 | 含义 |
|------|------|
| batch | 一次 CLI 调用处理的全部条目 |

**S-03 术语表**：
| 术语 | 含义 |
|------|------|
| batch | 一次存储调用写入的单条记忆 |

**评审意见**：
```text
🔴 阻断 | 跨文档 | S-01 和 S-03 对 "batch" 定义不同但父文档未标注
→ 修改建议：方案一：统一含义，在父文档 §4 共享术语速查中定义唯一含义；方案二：至少一方改名（如 S-03 改为 storeUnit），并在父文档 §4 标注两种语义
```

### ❌ 接口签名不匹配

**父文档 §4.2**：
```typescript
function importKb(source: string, options?: ImportOptions): Promise<ImportResult>;
```

**S-02 §4a.1**：
```typescript
function importKb(path: string, opts?: Options): Result;
```

**评审意见**：
```text
🔴 阻断 | S-02 §4a.1 | `importKb` 签名与父文档 §4.2 不一致（参数名、返回类型、同步/异步均不同）
→ 修改建议：对齐父文档签名，或更新父文档定义并同步所有引用方
```

### ❌ 一览图节点缺失

**一览图**：包含 S-01, S-02, S-03, S-04 四个节点

**实际文件**：仅有 S-01, S-02, S-03 三个子文档

**评审意见**：
```text
🟡 警告 | 父文档 §2 | 一览图标注 S-04 节点但无对应子文档
→ 修改建议：补充 S-04 子文档，或从一览图中移除 S-04 节点
```

## 七、评审报告反模式

### ❌ 模糊建议

```text
🟡 S-01 §3 方案设计建议优化
```

**问题**：没有说明优化什么、改成什么。

**正确写法**：
```text
🟡 S-01 §3 | 方案未说明并发场景下的行为，建议补充：当两个请求同时修改同一记录时，采用乐观锁还是悲观锁？
```

### ❌ 无依据挑战

```text
🔴 S-02 §3 Redis 方案不可靠
```

**问题**：没有说明为什么不可靠、在什么条件下不可靠。

**正确写法**：
```text
🔴 S-02 §3 | [置信度：高] Redis SETNX 在主从切换时可能丢失锁（主节点宕机未同步到从节点），若业务不允许短暂并发需改用 RedLock 或 etcd
```

### ❌ 越俎代庖

```text
🔴 S-01 §3 | 建议改用 etcd 实现分布式锁，具体方案如下：
1. 安装 etcd 3.5 集群
2. 使用 etcdctl lock 命令
3. ...
```

**问题**：给了完整替代方案，剥夺了设计者的决策权。

**正确写法**：
```text
🔴 S-01 §3 | [置信度：高] Redis SETNX 在主从切换时可能丢失锁，若业务不允许短暂并发需考虑更强一致性的锁方案（如 etcd/RedLock）
```

## 八、通用评审规则（降级模式）

当文档为自由格式时，使用以下通用规则替代 design-craft 基线：

### 结构检查

| 检查项 | 严重度 |
|--------|--------|
| 缺少背景/目标说明 | 🟡 |
| 缺少方案设计章节 | 🔴 |
| 缺少接口/数据模型定义 | 🟡 |
| 缺少异常处理说明 | 🟡 |
| 缺少风险/待定问题 | 🟢 |

### 内容检查

| 检查项 | 严重度 |
|--------|--------|
| 方案描述只有结论无推导 | 🟡 |
| 接口定义缺少参数或返回值 | 🔴 |
| 数据模型缺少字段含义 | 🟡 |
| 含空话（"众所周知"等） | 🟢 |
| 复制 PRD 原文未转化 | 🟡 |
