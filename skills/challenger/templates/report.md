# 质疑报告模板

## 报告结构

质疑报告应包含以下部分：

### 1. 复现流程确认

```markdown
## 📋 复现流程确认

### 复现场景
- **触发条件**：<触发条件描述>
- **复现步骤**：
  1. <步骤1>
  2. <步骤2>
  3. <步骤3>
  ...

### 预期与实际
- **期望行为**：<期望结果>
- **实际行为**：<实际结果>

### 确认状态
- ✅ 已确认 / ⚠️ 已修正 / ❌ 待确认
```

### 2. 变更概述

```markdown
## 📊 变更概述

- **变更类型**：<Bug修复/新增功能/优化>
- **变更描述**：<简要描述>
- **变更文件**：<文件列表>
- **变更目的**：<目的描述>
```

### 3. 质疑详情

```markdown
## 🔍 质疑详情

### 质疑 #1：<质疑标题>

- 🏷️ **类型**：<质疑类型>
- 📍 **涉及**：<代码位置>
- ❓ **质疑点**：<具体质疑内容>
- 🔍 **验证方法**：<如何验证这个质疑>
- ⚠️ **风险**：<如果质疑成立，可能的风险>
- 📊 **置信度**：<高/中/低>
- 💡 **建议**：<改进建议>

### 质疑 #2：<质疑标题>
...
```

### 4. 质疑总结

```markdown
## 📊 质疑总结

### 统计
- **总质疑数**：X
- **高风险**：X
- **中风险**：X
- **低风险**：X

### 风险分布
- 🔴 高风险：<风险描述>
- 🟡 中风险：<风险描述>
- 🟢 低风险：<风险描述>
```

### 5. 行动建议

```markdown
## 🎯 行动建议

### 必须处理
1. <行动项1> - <原因>
2. <行动项2> - <原因>

### 建议处理
1. <行动项1> - <原因>
2. <行动项2> - <原因>

### 可选处理
1. <行动项1> - <原因>
2. <行动项2> - <原因>
```

## 报告示例

```markdown
# 🎯 Bug修复质疑报告

## 📋 复现流程确认

### 复现场景
- **触发条件**：user_id在数据库中不存在
- **复现步骤**：
  1. 调用`get_user(user_id)`
  2. 数据库查询返回空结果
  3. 尝试访问返回对象的属性
  4. 抛出AttributeError

### 预期与实际
- **期望行为**：返回None或抛出自定义异常
- **实际行为**：抛出AttributeError

### 确认状态
- ✅ 已确认

## 📊 变更概述

- **变更类型**：Bug修复
- **变更描述**：修复get_user()函数未处理空值的问题
- **变更文件**：user.py, test_user.py
- **变更目的**：防止AttributeError异常

## 🔍 质疑详情

### 质疑 #1：根因深度验证

- 🏷️ **类型**：根因深度
- 📍 **涉及**：`user.py:get_user()` 第15行
- ❓ **质疑点**：为什么数据库查询会返回空？是数据不存在还是查询条件错误？
- 🔍 **验证方法**：检查user_id的来源和数据库查询逻辑
- ⚠️ **风险**：如果是查询条件错误，返回None可能掩盖了数据问题
- 📊 **置信度**：中
- 💡 **建议**：添加日志记录，监控空查询的频率和来源

### 质疑 #2：修复完整性验证

- 🏷️ **类型**：触发条件覆盖
- 📍 **涉及**：`user.py:get_user()` 第15-20行
- ❓ **质疑点**：是否只处理了None的情况？是否处理了空字符串、无效ID等情况？
- 🔍 **验证方法**：测试各种边界输入
- ⚠️ **风险**：未覆盖的边界情况可能导致新的bug
- 📊 **置信度**：中
- 💡 **建议**：添加输入验证，处理空字符串和无效ID

### 质疑 #2.5：流程覆盖验证

- 🏷️ **类型**：流程覆盖
- 📍 **涉及**：`user.py:get_user()` 整体逻辑
- ❓ **质疑点**：当前修复只针对user_id不存在的情况，如果user_id存在但数据异常（如字段为空、类型错误）呢？
- 🔍 **验证方法**：推理其他可能的触发流程并测试
- ⚠️ **风险**：修复可能只覆盖了当前流程，其他流程下仍有问题
- 📊 **置信度**：高
- 💡 **建议**：测试其他异常数据场景

**流程推理分析**：

基于当前复现流程：user_id不存在 → 数据库返回None → 访问属性抛出AttributeError

推理出的其他可能流程：

**流程A**：user_id存在但用户数据被删除
- 触发条件：用户数据被软删除或标记为无效
- 预期表现：查询返回空或返回无效数据对象
- 与当前流程的关系：类似但触发条件不同

**流程B**：user_id格式错误
- 触发条件：传入的user_id为空字符串、特殊字符、超长字符串
- 预期表现：查询异常或返回None
- 与当前流程的关系：输入层面的异常

**流程C**：数据库连接异常
- 触发条件：数据库连接超时、断开
- 预期表现：查询抛出连接异常
- 与当前流程的关系：系统层面的异常

**流程对比分析**：

| 流程 | 触发条件 | 修复前行为 | 修复后行为 | 是否有效 |
|------|----------|------------|------------|----------|
| 当前流程 | user_id不存在 | AttributeError | 返回None | ✅ |
| 流程A | 数据被删除 | AttributeError | 返回None | ✅ |
| 流程B | 格式错误 | 可能异常 | ❓未测试 | ❓ |
| 流程C | 连接异常 | 连接异常 | 连接异常 | ❌ |

结论：
- **有效覆盖**：当前流程和流程A
- **未覆盖**：流程B（输入验证）、流程C（异常处理）
- **风险**：输入验证和异常处理未覆盖

### 质疑 #3：语义影响验证

- 🏷️ **类型**：返回值变化
- 📍 **涉及**：`user.py:get_user()` 第20行
- ❓ **质疑点**：返回None是否改变了原有语义？调用方是否处理了None？
- 🔍 **验证方法**：检查所有调用方的处理逻辑
- ⚠️ **风险**：调用方未处理None可能导致新的bug
- 📊 **置信度**：高
- 💡 **建议**：更新调用方代码，处理None返回值

### 质疑 #4：同类问题验证

- 🏷️ **类型**：模式搜索
- 📍 **涉及**：`user.py:get_user_by_email()` 第25行
- ❓ **质疑点**：其他查询函数是否也有同样的问题？
- 🔍 **验证方法**：搜索所有类似的查询函数
- ⚠️ **风险**：只修复了一个函数，其他函数仍有问题
- 📊 **置信度**：高
- 💡 **建议**：批量修复所有类似的查询函数

### 质疑 #5：测试覆盖验证

- 🏷️ **类型**：测试覆盖
- 📍 **涉及**：`tests/test_user.py`
- ❓ **质疑点**：测试是否覆盖了user_id不存在的情况？
- 🔍 **验证方法**：检查测试用例
- ⚠️ **风险**：未覆盖的情况可能导致回归
- 📊 **置信度**：中
- 💡 **建议**：添加边界测试用例

## 📊 质疑总结

### 统计
- **总质疑数**：6
- **高风险**：3
- **中风险**：3
- **低风险**：0

### 风险分布
- 🔴 高风险：调用方未处理None、同类问题未修复、流程覆盖不全
- 🟡 中风险：根因未深入、边界情况未覆盖、测试不足

## 🎯 行动建议

### 必须处理
1. 更新调用方代码，处理None返回值 - 防止新的bug
2. 批量修复所有类似的查询函数 - 防止同类问题
3. 测试其他异常数据场景（输入验证、连接异常） - 确保流程覆盖

### 建议处理
1. 添加日志记录，监控空查询的频率和来源 - 帮助定位根因
2. 添加输入验证，处理空字符串和无效ID - 提高健壮性
3. 添加异常处理，处理数据库连接异常 - 提高系统稳定性

### 可选处理
1. 添加边界测试用例 - 防止回归
```

## 使用说明

1. **根据变更类型选择模板**：不同类型的变更使用不同的质疑策略
2. **复现流程必须确认**：没有复现的bug分析是没有意义的
3. **质疑必须有依据**：每个质疑必须有具体的验证方法
4. **风险必须评估**：每个质疑必须评估风险等级
5. **建议必须具体**：改进建议必须具体可执行
