# Implementation Report（实现报告）

需求完成后，生成实现总结报告，记录"最终做成了什么"。

## 定位

```
requirement-mining → ... → 开发完成 → implementation-report
      需求摘要                           实现报告（一一对应）
```

- **输入**：需求摘要文件 + 设计文档 + 代码变更
- **输出**：实现总结报告
- **边界**：只记录实现结果，不参与开发过程

## 核心原则

1. **与需求一一对应**：每份实现报告对应一份需求摘要
2. **结论导向**：记录最终结果，不记录开发过程
3. **可追溯**：通过 commits 字段关联代码变更
4. **自动配置**：使用 `.requirements/config` 中的存储路径

## 存储位置

检查项目中是否已配置存储位置（`.requirements/config`）：

- **已配置**：读取 `storage_path`，报告存放在需求目录内 `{storage_path}/{YYYY-MM-DD}-{feature-name}/report.md`
- **未配置**：询问用户，给出默认建议 `.requirements/`

## 触发条件

| 触发方式 | 说明 |
|----------|------|
| 自动触发 | 需求状态变为"已完成"时 |
| 手动触发 | 用户说"生成实现报告"、"记录完成情况" |

## 工作流

### 1. 读取需求摘要

从需求目录 `{storage_path}/{YYYY-MM-DD}-{feature-name}/requirement.md` 中提取：
- 核心诉求
- 需求清单（需求 ID、描述、验收标准）
- 关键假设
- 非功能性约束

### 2. 读取设计文档

从需求目录的 `design/` 子目录中找到设计文档，提取：
- 关键设计决策
- 接口设计
- 数据模型

### 3. 收集代码变更

通过 git log 收集与本次需求相关的提交：
- 用户手动提供 commit hash，或
- 根据需求时间范围自动搜索

### 4. 生成报告

```markdown
---
feature: {功能名称}
created: {YYYY-MM-DD}
type: implementation-report
---

# 实现报告：{功能名称}

## 1. 需求回顾
（引用需求摘要的核心诉求，一句话）

## 2. 设计概要
（引用设计文档的关键决策，不复制全文）

## 3. 实现结果
| 需求 ID | 需求描述 | 实现状态 | 说明 |
|--------|----------|----------|------|
| REQ-01 | ... | ✅ 已实现 | ... |
| REQ-02 | ... | ⚠️ 部分实现 | 原因... |
| REQ-03 | ... | ❌ 未实现 | 原因... |

## 4. 偏差记录
| 偏差项 | 设计方案 | 实际实现 | 原因 |
|--------|----------|----------|------|
| ... | ... | ... | ... |

（如无偏差，写"无偏差"）

## 5. 关联提交
| 提交 | 说明 |
|------|------|
| abc1234 | ... |
| def5678 | ... |

## 6. 验收对照
| 验收标准 | 是否满足 | 说明 |
|----------|----------|------|
| ... | ✅/❌ | ... |

## 7. 遗留问题
（如有未完成的子项或已知问题）

（如无遗留，写"无遗留问题"）
```

### 5. 更新需求摘要

报告生成后，更新需求摘要文件的元数据：

```yaml
---
feature: 用户注册
created: 2026-06-08
status: 已完成
updated: 2026-06-10
commits:
  - abc1234
  - def5678
report: .requirements/2026-06-08-user-registration/report.md
---
```

### 6. 输出结果

```text
✅ 实现报告已生成

📄 {storage_path}/{YYYY-MM-DD}-{feature-name}/report.md

需求摘要已更新：
- status: 已完成
- commits: [提交列表]
- report: [报告路径]
```

## 反模式

- ❌ 报告写成开发日志 → 只记录最终结果，不记录过程
- ❌ 遗漏偏差记录 → 与设计不一致的地方必须记录
- ❌ commits 不准确 → 必须确认提交与需求的对应关系
- ❌ 未更新需求摘要 → 报告生成后必须回写需求文件的元数据
