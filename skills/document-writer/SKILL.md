# 项目文档生成器

## 何时使用

满足以下任一条件时触发本 skill：

- 用户明确要求为项目生成 README 或完整文档
- 新建项目需要对外发布的文档
- 已有项目缺少文档需要补齐
- 用户说"写个 README"、"生成文档"、"document this"、"补文档" 等

### 何时不使用

以下情况**不触发**本 skill：

- 用户只是问"README 怎么写" → 直接回答，不需要走完整流程
- 用户只要求写单一章节（如只写 License 部分）→ 直接写，不走全量流程
- 项目没有代码（纯文档仓库、配置仓库）→ 无法自动识别类型，直接帮助写
- 用户明确表示只需要一个简单草稿 → 不走质量检查流程

### 快速通道

项目同时满足以下**全部条件**时，跳过阶段 2（策略确认），合并阶段 1 + 2，直接进入内容生成：

- 公开 API 或命令数 ≤ 5 个
- 配置项 ≤ 5 个
- 用户未要求子文档拆分
- 项目类型判定明确（非混合型）

快速通道只生成单个 README（不拆子文档），但仍遵守全部质量底线。

## 核心原则

1. **类型驱动策略**：先识别项目类型，再套用对应策略。不用一套模板写所有项目
2. **示例即血肉**：每个对外接口（API/命令/配置项）必须有输入→输出完整示例，这是不可妥协的底线
3. **README 是索引**：README 不超过一屏，做导航枢纽；详细信息按主题拆分到子文档
4. **按需拆分**：有拆分阈值，简单项目不冗余拆分，复杂项目不遗漏文档
5. **零空洞容忍**：禁止出现 "TODO"、"Coming soon"、"See source code" 等占位符

## 工作流总览

```
阶段 1：项目扫描 + 类型识别   → 分析项目结构，自动判定类型，允许手动覆盖
阶段 2：策略确认 + 拆分决策   → 根据类型确定文档集，阈值判断是否需要子文档
阶段 3：内容生成              → 按策略模板逐章节生成，严格遵守示例规则
阶段 4：质量检查              → 扫描空洞内容、验证示例完整性、检查链接有效性
阶段 5：输出落盘              → 写入文件，给出文档导航概览
```

## 阶段 1：项目扫描 + 类型识别

### 1.1 扫描项目结构

分析以下内容（扫描结果不足以判断时再询问用户）：

- 根目录文件：`package.json`、`setup.py`、`pyproject.toml`、`go.mod`、`Cargo.toml`、`Makefile` 等
- 目录结构：`src/`、`lib/`、`commands/`、`routes/`、`controllers/` 等
- 入口文件：`index.js`、`main.go`、`app.py`、`cli.js` 等
- 依赖框架：从依赖列表中识别 Express、Flask、React、Commander 等
- 现有文档：是否已有 README、CONTRIBUTING、CHANGELOG 等

### 1.2 类型判定规则

对比以下特征判定项目类型。详细策略见 `references/strategies.md`。

| 项目类型 | 核心判定特征 |
|----------|-------------|
| **库 / SDK** | `package.json` 含 `main`/`exports`/`module`；`setup.py` 配置；Go module；主要导出函数/类供其他项目引用 |
| **CLI 工具** | `package.json` 含 `bin` 字段；使用 commander/yargs/cobra/click/argparse 等 CLI 框架；入口文件以解析命令行参数为主要逻辑 |
| **Web 应用/服务** | 依赖 express/koa/next/flask/django/gin 等 Web 框架；含路由/控制器定义；包含 HTTP 服务启动逻辑 |
| **插件/扩展** | 依赖宿主框架（VS Code/Webpack/Babel/ESLint 等）；`package.json` 含 `engines`/`contributes`/`activationEvents` |

**混合类型处理**：若项目同时满足多种类型特征（如 CLI + 库），判定为「混合型」，合并对应策略。输出判定结果让用户确认。

### 1.3 逆向生成模式

若项目**完全没有 README**，进入逆向模式：
- 从代码入口文件反推项目用途
- 从导出接口反推 API 列表
- 从 CLI 入口反推命令列表
- 标注「从代码推断，请验证」的提示

### 1.4 已有 README 更新模式

若项目**已有 README 但质量不达标**，进入更新模式而非全量重写：

- **保留**：项目描述、已有徽章、用户确认过的准确信息
- **补充**：缺失的示例、空白章节、占位符内容
- **新增**：符合该类型策略的子文档（若触发拆分阈值）
- **重写**：信息密度不达标的章节（概述只有一句话、无输出的命令等）

输出判定结果时标注哪些章节是保留、哪些是新增/重写，让用户确认。

### 1.5 输出判定结果

```markdown
## 项目类型判定

- **项目名称**：[从 package.json/setup.py 等提取]
- **判定类型**：[库 / CLI / Web 应用 / 插件 / 混合型]
- **判定依据**：[列出匹配到的关键特征]
- **推荐文档集**：[列出将生成的文档列表]

请确认或手动指定类型。
```

## 阶段 2：策略确认 + 拆分决策

### 2.1 获取策略模板

根据确认的类型，从 `references/strategies.md` 中加载对应策略模板。每个策略模板定义了：

- README 各章节（标题 + 内容要求 + 必填/可选）
- 子文档列表（名称 + 触发条件 + 内容要求）
- 各章节的示例强制规则

### 2.2 拆分阈值判断

**库/SDK 类型**：

| 触发条件 | 拆分子文档 |
|----------|-----------|
| 公开 API（函数/类/方法）> 5 个 | `api-reference.md` |
| 配置项 > 10 个 | `configuration.md` |
| 有高级用法（中间件/插件/钩子/生命周期） | `advanced-usage.md` |

**CLI 类型**：

| 触发条件 | 拆分子文档 |
|----------|-----------|
| 命令数 > 5 个 | `command-reference.md` |
| 配置项 > 10 个 | `configuration.md` |
| 支持 CI/CD 集成或脚本调用 | `integration.md` |

**Web 应用类型**：

| 触发条件 | 拆分子文档 |
|----------|-----------|
| 部署步骤 > 3 步 或 多环境 | `deployment.md` |
| 模块/服务 > 3 个 | `architecture.md` |
| 有本地开发特殊配置 | `development.md` |

**插件类型**：

| 触发条件 | 拆分子文档 |
|----------|-----------|
| 配置项 > 10 个 | `configuration.md` |
| API/Hooks > 5 个 | `api-reference.md` |
| 有多个使用场景示例 | `examples.md` |

**通用规则**：若项目有 `CONTRIBUTING.md`、`CHANGELOG.md`、`LICENSE` 等标准文件，也纳入文档集。

### 2.3 确认文档集

```markdown
## 文档生成计划

### 主文档
- `README.md` — 索引枢纽，包含概述、快速开始、子文档导航

### 子文档（共 N 个）
- `api-reference.md` — [触发原因：API 数量 12 > 阈值 5]
- `configuration.md` — [触发原因：配置项 15 > 阈值 10]
- ...

### 不拆分的文档
- [说明哪些未触发阈值，内容并入 README 对应章节]

确认后开始生成内容。
```

## 阶段 3：内容生成

### 3.1 内容质量底线

严格遵守 `references/quality-rules.md` 中定义的全部禁止模式（Q-01 至 Q-06）和完整性检查规则。核心要求：

- 每个对外接口必须有输入→输出完整示例
- 每个命令必须附带预期终端输出
- 禁止任何形式的占位符和模糊表述
- 章节要么写满实质内容，要么从导航中移除

### 3.2 README 生成规则

README 作为索引枢纽，遵循以下结构约束：

**通用结构**（所有类型）：

| 章节 | 要求 |
|------|------|
| 标题 + 徽章 | 项目名 + 一行描述 + 可选徽章（npm/CI/coverage） |
| 概述 | 做什么 + 解决什么问题 + 与其他方案的简要区别 |
| 快速开始 | 最简安装 + 最简示例（输入 + 输出），让读者 30 秒内理解项目价值 |
| 子文档导航 | 每个子文档链接 + 一句话说明「这个文档里有什么」 |

**类型专属章节**（根据策略模板追加在「快速开始」之后）：

| 类型 | 追加章节 |
|------|----------|
| 库/SDK | 核心 API 速览（精选 3-5 个最常用 API，各附一句示例） |
| CLI 工具 | 核心命令速览（精选 3-5 个最常用命令，各附终端输出片段） |
| Web 应用 | 架构速览（一张文字拓扑图，说明模块关系） |
| 插件 | 最简配置示例（最小化配置即可跑通的例子） |

**README 长度控制**：控制在 60-100 行以内。超过 120 行，说明有内容应该下沉到子文档。

### 3.3 子文档生成规则

每个子文档独立可读，不依赖 README 上下文。结构：

```
# {子文档标题}

## 概述
[该文档覆盖什么内容，与 README 的关系]

## {核心内容}
[按策略模板填充，确保每节有实质内容]

## 相关文档
[回到 README 和其他子文档的链接]
```

子文档间内容禁止重复。若同一信息需要出现在多个文档，在一个文档写全，其他文档用链接引用。

### 3.4 示例生成策略

示例是文档的灵魂。按以下优先级获取示例：

1. **从测试文件提取**（最可靠）：搜索 `test/`、`__tests__/`、`*_test.go` 等，提取真实调用示例
2. **从入口/示例目录提取**：搜索 `examples/`、`demo/`、`sample/` 目录
3. **从代码逻辑推断**：分析接口签名和实现，构造合理示例。标注 `<!-- 示例从代码推断，请验证 -->`
4. **标注不可得**：若上述途径均无法获取，标注「暂无可用示例，欢迎贡献」

## 阶段 4：质量检查

生成完成后，逐条检查：

### 4.1 自动检查清单

- [ ] 全文无 Q-01 到 Q-06 的禁止模式
- [ ] 每个对外接口有完整示例（输入+输出）
- [ ] 所有子文档链接可对应到实际生成的文件
- [ ] README 行数 ≤ 120
- [ ] 子文档间无重复内容（抽查相邻文档）
- [ ] 概述章不只是标题扩充版

### 4.2 修正缺陷

若检查发现缺陷，立即修正后重新检查，直到全部通过。

## 阶段 5：输出落盘

### 5.1 写入文件

将生成的文档写入项目根目录。若用户指定了输出目录，写入指定路径。

### 5.2 输出导航概览

```markdown
## 文档生成完成

### 已生成文档

| 文件 | 内容摘要 | 行数 |
|------|----------|------|
| README.md | 项目概述 + 快速开始 + 导航 | 85 |
| api-reference.md | 12 个 API 的完整参考 | 340 |
| configuration.md | 15 个配置项详解 | 120 |

### 快速验证

- 打开 `README.md` → 快速开始 → 复制示例命令 → 应可直接跑通
- 打开 `api-reference.md` → 任意 API → 有明确的输入输出示例
- 子文档间链接 → 全部可达，无死链
```
